What is retryWhen?

The retryWhen operator provides fine-grained control over retry logic. Unlike retry, it allows you to implement custom retry strategies like exponential backoff, conditional retries, and delayed retries based on error types.

When to Use

  • ✅ Implement exponential backoff strategy
  • ✅ Retry only specific error types
  • ✅ Add delays between retry attempts
  • ✅ Complex retry logic with conditions

How It Works

1

Error Notification

When an error occurs, retryWhen receives it as a notification

2

Custom Logic

Your custom function decides whether and when to retry

3

Delay Strategy

Apply delays (exponential backoff, fixed, etc.) before retry

4

Resubscribe or Error

Either resubscribe to source or propagate the error

Real-World Use Cases

📡 API Rate Limiting

Exponential backoff for rate-limited APIs

retryWhen(errors => errors.pipe(delay(2 ** retryCount * 1000)))

🔐 Authentication Retry

Retry only on 401 errors

retryWhen(errors => errors.pipe(filter(e => e.status === 401)))

🌐 Network Recovery

Wait for network before retrying

retryWhen(errors => errors.pipe(delayWhen(() => networkOnline$)))

⏰ Time-based Retry

Retry during business hours only

retryWhen(errors => errors.pipe(delayWhen(() => waitForBusinessHours())))

💡 Retry Strategies Explained

🚀 Exponential Backoff

Delay increases exponentially: 1s, 2s, 4s, 8s...

delay = 2^attempt × 1000ms

Best for: Rate-limited APIs, server overload

⏱️ Fixed Delay

Same delay between all retries: 1s, 1s, 1s...

delay = 1000ms

Best for: Predictable retry timing

⚡ Immediate

No delay between retries

delay = 0ms

Best for: Quick transient errors

⚠️ Important Notes

  • Complex operator: Requires understanding of higher-order Observables
  • Must emit to retry: The notifier Observable must emit to trigger retry
  • Error propagation: If notifier errors or completes, retry stops
  • Memory consideration: Long retry chains can accumulate memory

✅ Best Practices

  • Use exponential backoff for production systems
  • Set maximum retry limits to prevent infinite loops
  • Log retry attempts for monitoring and debugging
  • Consider error types when deciding to retry
  • Add jitter to prevent thundering herd problem
